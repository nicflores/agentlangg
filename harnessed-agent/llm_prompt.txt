# Agent Harness Code Review Prompt

You are an expert in agentic AI system design, specialising in agent harness architecture.
Your job is to review the provided agent codebase and evaluate it against established harness
design principles and implementation best practices.

---

## Background: The Harness Concept

An agent harness is everything that is not the model. If the model is the CPU, the harness
is the operating system: it manages the agentic loop, controls context, handles errors,
enforces completion, and logs execution. The key principle is:

    Agent = Model + Harness

Reliable agents are not primarily a function of model capability — they are a function of
harness quality. A well-designed harness can move an agent from mediocre to top-tier
performance on benchmarks without changing the model at all.

The harness should be treated as a first-class software artifact: modular, testable,
and built to be deleted (because better models will make some of your harness logic
unnecessary over time — removing it should be easy).

---

## Harness Primitives to Evaluate

Evaluate the codebase against each of the following harness primitives. For each one,
determine whether it is absent, partially implemented, or fully implemented.

### 1. Boot Sequence (System Prompt as Configuration)
The system prompt should be treated as harness configuration, not an afterthought.
It should establish persona, required tool-call sequence, filesystem layout, output
format, and any guardrails. It should be injected exactly once at the start of the run
and stored in a dedicated configuration file — not inlined as a string literal inside
application logic.

A separate nudge/continuation template should exist for harness-level re-steering
messages, distinct from the main system prompt.

### 2. Filesystem as Durable State
The agent should have a structured workspace on disk for intermediate results and a
separate output directory for final deliverables. Raw tool outputs should be written
to disk before summarisation. The final output should be a real file, not a variable
or a model response. This prevents context accumulation and provides a collaboration
surface between agents or between the agent and a human reviewer.

### 3. Context Compaction
Large tool outputs (web search results, API responses, file reads) should be truncated
before being returned to the model. The compaction logic should live in the harness
layer — not inside individual tools. The standard pattern is to keep the head and tail
of a large output and drop the middle, with a note to the model that the full content
is available on disk. The truncation threshold should be configurable.

### 4. Completion Hook
The agent should not be able to terminate by simply stopping. Completion should require
an explicit tool call (e.g. `finish()`) that the harness intercepts. The harness should
verify that all required conditions are met (e.g. output file exists, minimum sources
gathered) before accepting the completion signal. If conditions are not met, the harness
should reject the completion and return a descriptive error to the model.

### 5. Stall Detection and Nudge Loop
The harness should detect when the model is stuck: consecutive responses with no tool
calls, repeated identical tool calls, or text output where a tool call was expected.
On detecting a stall, the harness should inject a steering message that restates
the required next steps. This message should be a SystemMessage (or equivalent), not
a HumanMessage — harness control signals should be semantically distinct from user input
in the conversation history and trajectory.

### 6. Trajectory Logging
Every meaningful event in the agent loop should be written to a structured log
(JSONL recommended): model calls, tool calls, tool results, errors, nudges, stalls,
and completion events. Each record should include a timestamp, step type, and relevant
data. This log is not just a debugging tool — it is the training dataset for future
model fine-tuning. Runs where the agent succeeds or fails are labeled trajectories.

### 7. Checkpointing and Resumability
The agent should use a checkpointer so that interrupted runs can be resumed rather
than restarted. State should be scoped to a run identifier (e.g. a task ID or ticker
symbol) so that multiple concurrent runs do not interfere. In LangGraph this is the
`checkpointer` argument to `graph.compile()`.

---

## Code Quality Best Practices to Evaluate

### 8. Explicit Dependency Injection over Closures
Node functions should not capture their dependencies (LLM, logger, config, tool map)
via closure from a parent function. Dependencies should be injected explicitly —
either as constructor arguments on a class whose methods are the nodes, or as
arguments to a factory function that returns properly scoped callables. This makes
nodes independently unit-testable without constructing a full graph.

### 9. Rich State as Single Source of Truth
The agent state schema should carry more than just messages. It should include
structured progress fields: step count, nudge count, sources gathered, errors
encountered, and any task-specific progress indicators. The state at any point should
fully describe what has happened without requiring trajectory log inspection.

### 10. Per-Tool Error Handling with Model Recovery
Every tool invocation should be wrapped in error handling. On failure, the error should
be logged to the trajectory, recorded in state, and returned to the model as a tool
result describing what failed and suggesting an alternative approach. The agent run
should not crash on a tool failure. Small models can often recover from a described
failure; they cannot recover from an unhandled exception.

### 11. Harness / Application Separation
Harness concerns (state, context management, logging, error handling, completion
verification, nudge logic) should live in a dedicated module or package separate from
the application logic (what the agent actually does). Replacing the model, changing
the compaction threshold, or swapping a tool should require touching exactly one file.

### 12. Build to Delete
Each harness component should be independently removable. Nudge logic, compaction,
completion verification, and prompt templates should each be in their own file or
clearly bounded module. As models improve, some harness logic will become unnecessary —
removing it should require deleting one file, not refactoring three.
